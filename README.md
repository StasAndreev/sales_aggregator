# Sales Aggregator

REST API сервис для загрузки, хранения и агрегации данных о продажах с маркетплейсов Ozon, Wildberries и Яндекс Маркет.

## Возможности

- Пакетная загрузка продаж через JSON или CSV-файл
- Получение списка продаж с фильтрацией и пагинацией
- Агрегированные метрики с группировкой по маркетплейсу / дате / статусу
- Конвертация в USD по курсу ЦБ РФ с кэшем на 1 час
- Топ-N продуктов по выручке, количеству или прибыли

## Стек технологий

- Python 3.12, FastAPI, Pydantic v2
- SQLite через стандартный `sqlite3`
- Pandas
- pytest

## Локальный запуск

```bash
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

uvicorn main:app --reload
```

Документация API: http://127.0.0.1:8000/docs

## Docker

```bash
docker build -t sales-aggregator .
docker run -p 8000:8000 sales-aggregator
```

> БД хранится внутри контейнера и сбрасывается при перезапуске.

## Обзор API

| Метод  |            Путь           |                    Описание                        |
|--------|---------------------------|----------------------------------------------------|
| `POST` | `/sales`                  | Добавить одну или несколько продаж (batch)         |
| `GET`  | `/sales`                  | Список продаж с фильтрами и пагинацией             |
| `POST` | `/sales/upload-csv`       | Загрузка продаж из CSV-файла (макс. 10 МБ)         |
| `GET`  | `/analytics/summary`      | Агрегированные метрики за период                   | 
| `GET`  | `/analytics/summary-usd`  | Те же метрики, конвертированные в USD              |
| `GET`  | `/analytics/top-products` | Топ-N продуктов по выручке, количеству или прибыли |

Пары `(order_id, marketplace)` ожидаются уникальными, дубликаты при вставке молча игнорируются.

## Заметки
1. С учётом размера БД тестовая БД создаётся для каждого теста. С расширением масштаба БД может потребоваться другой подход к тестированию.
2. Предположил, что order_ID + marketplace должны быть уникальные, для предотвращения дупликатов данных. Дупликаты игнорируются (чтобы не перегружать код задачи).
3. Если размер БД превышает размер SQL, нужно будет добавить пагинацию для рассчёта метрик.
4. Метрики рассчитываются по запросу, т.к. сервис предполагается небольшого размера
5. Добавлен индекс в БД, с предположением о том, что:
    - продажи чаще добавляются с самыми поздними датами, и добавление в индекс эффективно
    - удаление данных не осуществляется, поэтому индекс стареет очень медленно
6. Стоимость в БД хранится как текст для сохранения точности, т.к. через SQL не проводятся арифметические операции.
